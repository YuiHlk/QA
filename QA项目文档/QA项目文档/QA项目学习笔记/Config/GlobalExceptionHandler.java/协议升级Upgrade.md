## 核心结论：

HTTP 本身是**短连接、一问一答、单向请求模型**，天生不适合双向持续通信；WebSocket 是一套全新独立协议，没法直接在 TCP 上凭空建立，只能借 HTTP 握手完成切换，这个切换动作就叫**协议升级（101 Switching Protocols）**。
## 一、HTTP 的底层短板：

HTTP/1.1 设计初衷就是**单次请求 - 单次响应**：

1. 客户端发请求 → 服务端完整返回数据 → 连接就算完成，随时可以断开；
2. 报文结构完全是「请求头 + 请求体」「响应头 + 响应体」，服务端不能主动随便发消息；
3. HTTP 报文有严格边界、长度标识，无法自由、持续双向收发零碎消息。

如果直接在 TCP 上用 HTTP 做双向实时聊天会有一堆问题：

- 服务端不能主动推送消息给客户端；
- 每次发数据都要重复带上一整套 HTTP 请求 / 响应头，冗余极大；
- 没有统一的消息帧格式，分不清文本、二进制、关闭、心跳包；
- 浏览器同源、安全策略没有标准化校验方式。

所以业界单独设计了 **WebSocket 二进制帧协议**，专门用于低开销双向实时通信。

但浏览器不能直接裸 TCP 连接（浏览器安全策略禁止任意 TCP 套接字），必须依托 HTTP 通道完成握手。

## 二、协议升级完整逻辑：借 HTTP 搭个桥

WebSocket 不能绕过 HTTP，流程分为两段：

### 阶段 1：标准 HTTP 握手（用来协商升级）

浏览器发起 GET 请求，带上专属升级标识头：
```http
GET /ws HTTP/1.1
Host: shturl.
Upgrade: websocket        // 告诉服务器：我要切换协议
Connection: Upgrade       // 配套标识，代表本次连接要升级
Sec-WebSocket-Key: 随机密钥 // 安全校验，防止伪造连接
Sec-WebSocket-Version: 13
```

这一步本质还是普通 HTTP 请求，网关、Nginx、防火墙都能识别，可以正常做鉴权、限流、跨域校验。

### 阶段 2：服务器同意升级，返回 101

服务器校验 Key、版本合法后，返回特殊状态码 `101 Switching Protocols`：
```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: 密钥加密结果
```

`101` 的含义：**放弃当前 HTTP 协议，切换为 Upgrade 指定的新协议**。

### 阶段 3：TCP 通道彻底脱离 HTTP，使用 WebSocket 私有帧

101 返回之后，**这条 TCP 连接不再遵守 HTTP 任何规范**：

- 不再有 Request/Response 头；
- 所有数据都用 WebSocket 自定义**帧格式**传输；
- 支持双向随时发消息、区分文本 / 二进制、心跳、关闭帧；
- 头部开销极小，只有几个字节，比 HTTP 轻量几十倍。
## 三、为什么不能像 SSE 一样，直接用普通 HTTP 长连接？

SSE 妥协了：只做**服务端单向推送**，全程保留 HTTP 格式，牺牲双向能力换取简单。

而 WebSocket 目标是**双向自由通信**，HTTP 结构会严重拖累性能：

1. 每次服务端主动发消息，都要拼接一整套 HTTP 响应头（Content-Type、Cache-Control 等），带宽浪费严重；
2. HTTP 没有标准分片帧标识，无法区分多条独立消息，粘包拆包需要自己处理；
3. HTTP 不原生支持二进制传输；
4. 无法统一规范心跳、异常关闭、消息分片。

想要低成本双向通信，就必须抛弃 HTTP 报文格式，换成独立协议，也就必须经过「升级切换」。

## 四、协议升级的额外作用（安全层面）

1. **身份校验**
    
    升级握手阶段还是 HTTP，可以通过 Cookie、Token、请求头完成登录校验；协议切换后不再有 HTTP 头，没法再做常规鉴权。
2. **防伪造跨域攻击**
    
    `Sec-WebSocket-Key` 密钥校验机制，避免恶意脚本伪造 WebSocket 连接。
3. **中间件兼容过渡**
    
    老旧防火墙、代理只识别 HTTP 流量，不识别裸 TCP 二进制流。先以 HTTP 握手通行，再切换协议，提升网络兼容性。
