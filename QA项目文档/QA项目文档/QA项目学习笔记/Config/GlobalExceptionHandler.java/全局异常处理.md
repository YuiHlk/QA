## 概述

`GlobalExceptionHandler` 是 QA 项目中的全局异常处理器，位于 `com.test.qa.config` 包下。基于 Spring 的 `@RestControllerAdvice` 注解实现，核心职责是**统一兜底所有未处理异常，避免堆栈信息直接暴露给前端**。

## 核心代码

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(AsyncRequestTimeoutException.class)
    public void handleAsyncTimeout(AsyncRequestTimeoutException e, HttpServletResponse response) {
        if (response.isCommitted()) {
            log.warn("SSE 异步请求超时，连接已关闭");
        }
    }

    @ExceptionHandler(IllegalArgumentException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public Result<Void> handleIllegalArgument(IllegalArgumentException e) {
        log.warn("参数异常: {}", e.getMessage());
        return Result.error(400, e.getMessage());
    }

    @ExceptionHandler(RuntimeException.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public Object handleRuntime(RuntimeException e, HttpServletResponse response) {
        if (isSseResponse(response)) {
            log.warn("SSE流异常（跳过JSON错误返回）: {}", e.getMessage());
            return null;
        }
        log.error("运行时异常", e);
        return Result.error(500, "服务器内部错误: " + e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public Object handleUnknown(Exception e, HttpServletResponse response) {
        if (isSseResponse(response)) {
            log.warn("SSE流异常（跳过JSON错误返回）: {}", e.getMessage());
            return null;
        }
        log.error("未知异常", e);
        return Result.error(500, "服务器内部错误");
    }

    private boolean isSseResponse(HttpServletResponse response) {
        if (response.isCommitted()) {
            return true;
        }
        String contentType = response.getContentType();
        return contentType != null && contentType.contains("text/event-stream");
    }
}
```

## 设计模式：分层捕获 + 响应类型感知

采用四个 `@ExceptionHandler` 方法按异常类型的 specificity 从窄到宽排列，Spring 自动匹配最精确的 handler。

### 异常捕获层级

```
AsyncRequestTimeoutException  →  handleAsyncTimeout()      (最精确)
IllegalArgumentException      →  handleIllegalArgument()    (参数校验)
RuntimeException              →  handleRuntime()            (运行时异常)
Exception                     →  handleUnknown()            (兜底)
```

### 返回类型差异

| Handler                   | 返回类型       | 原因                                 |
| ------------------------- | -------------- | ------------------------------------ |
| `handleAsyncTimeout`      | `void`         | 响应已提交，无需返回                 |
| `handleIllegalArgument`   | `Result<Void>` | 普通请求，返回结构化 JSON            |
| `handleRuntime`           | `Object`       | 需根据 SSE 判断返回 `null` 或 Result |
| `handleUnknown`           | `Object`       | 同上                                 |

## 设计要点详解

### 1. SSE 响应类型感知

`handleRuntime()` 和 `handleUnknown()` 在处理前先调用 `isSseResponse()` 判断当前响应是否为 [[SSE流式连接（Server-Sent Events）]]。

**为什么要这样做？** Spring MVC 的 `HttpMessageConverter` 无法将 `Result` 对象序列化为 `text/event-stream` 格式。如果强行返回 `Result`，会抛出 "No converter for Result with Content-Type 'text/event-stream'" 的二次异常，污染日志并导致连接处理异常。

因此对于 SSE 响应，直接返回 `null`，交由 Tomcat 自行关闭连接。

**`isSseResponse()` 的判断逻辑：**

1. 响应已提交（committed）→ 无法再修改响应内容，直接返回 `true`
2. 响应 Content-Type 包含 `text/event-stream` → 判定为 SSE

两层判断是递进关系：committed 意味着木已成舟，连 Content-Type 都不用看了。

### 2. AsyncRequestTimeoutException 的特殊处理

这是 SSE 场景下的典型异常——当 `SseEmitter` 异步请求超时时，Spring 抛出此异常。此时响应通常已经提交，不能再写任何内容，所以 handler：

- 返回 `void`，不做任何输出
- 只记录一条 `warn` 级别日志
- 内部 `response.isCommitted()` 的 if 判断是一次**防御性检查**，理论上进入此 handler 时响应已经提交，但加上判断更安全

### 3. 日志级别的分级策略

| 异常类型                           | 日志级别    | 是否记录堆栈 | 原因               |
| ------------------------------ | ------- | ------ | ---------------- |
| `IllegalArgumentException`     | `warn`  | 否      | 客户端输入问题，非系统故障    |
| `AsyncRequestTimeoutException` | `warn`  | 否      | 预期的连接生命周期事件      |
| `RuntimeException`             | `error` | 是      | 服务端问题，需要关注并排查    |
| `Exception`                    | `error` | 是      | 未预料到的错误，需要完整堆栈定位 |

这种分级策略确保运维人员看到 `error` 日志时可以立即确认是需要响应的系统问题，而不会被客户端参数错误干扰。

### 4. 异常信息暴露的控制

**`handleUnknown()`** 对外返回固定的「服务器内部错误」消息，不携带任何异常细节——这是正确的做法，因为未知异常可能包含敏感的内部路径、SQL 片段等。

**`handleRuntime()`** 通过 `e.getMessage()` 拼接了异常消息，在生产环境中这有一定风险。比如某个 RuntimeException 的 message 可能包含数据库连接信息或内部类路径。更严格的做法是与 `handleUnknown()` 一样返回固定消息，或将异常详情写入日志、返回一个通用的错误码让开发人员通过日志追踪。

## 与 Result 类的协作

`Result<T>` 是项目的统一响应结构：

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Result<T> {
    private int code;
    private String message;
    private T data;

    public static <T> Result<T> error(int code, String message) {
        return new Result<>(code, message, null);
    }
}
```

全局异常处理器通过 `Result.error(500, "...")` 快速构造错误响应，保证异常场景下的 JSON 结构与正常响应一致，前端只需要解析一种数据结构。

## 整体评价

这是一个简洁、实用的全局异常处理设计。最突出的亮点是**对 SSE 流式响应的兼容处理**——这类问题在实际项目中很容易被忽略，一旦 SseEmitter 超时或发生异常，如果没有 `isSseResponse()` 的判断分支，日志中会充斥 "No converter for Result with Content-Type 'text/event-stream'" 的错误，既干扰排障又无实际意义。

可改进的方向：`handleRuntime()` 对外暴露了 `e.getMessage()`，建议生产环境做脱敏处理。
