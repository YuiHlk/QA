## 项目概述

这是一个 **RAG（检索增强生成）问答质量评估与优化平台**，基于 Spring Boot + MyBatis-Plus + Spring AI 构建。核心目标是构建一个从知识库构建 → RAG问答 → 自动化评测 → 消融实验 → 模型微调的完整优化闭环。

- **技术栈**：Java 21 + Spring Boot 3.x + MyBatis-Plus + Spring AI + Ollama + ChromaDB + MySQL
- **架构分层**：Controller → Service（接口+实现）→ Mapper → Domain，统一返回 `Result<T>` 结构

### 核心实现逻辑

**1. 文档上传 (DocumentServiceImpl.upload)**

```
接收文件 → 保存到本地磁盘(uploads目录) → 创建RagDocument记录(状态=PROCESSING)
→ 注册事务同步回调(afterCommit) → 事务提交后触发异步处理
```

关键技术点：使用 `TransactionSynchronization.afterCommit()` 确保异步处理线程能在事务提交后读到刚插入的文档记录。如果不用这个机制，`@Async` 的线程池可能在事务提交前就执行了。

**2. 异步文档处理 (DocumentServiceImpl.processDocument)**

```
@Async 异步执行:

加载文档记录 → 提取文本(parseText)
→ TextChunkService.chunk() 滑动窗口分块(默认512字符/块，重叠64)
→ EmbeddingService.embedAndStore() 
   ├── 逐块调用 Ollama /v1/embeddings 生成向量(默认模型bge-m3)
   └── 批量 POST ChromaDB /api/v2/.../collections/{id}/add 存储向量
→ 每块生成RagChunk记录(chunkEmbeddingId关联ChromaDB)
→ 更新文档状态为 COMPLETED 或 FAILED
```

**3. 删除文档**

级联删除关联的 `RagChunk` 记录（MySQL层面）。ChromaDB 中的向量数据目前没有同步删除。

### 支撑服务说明

- **TextChunkService**：纯内存操作，滑动窗口策略切分文本。带边界校验（128~4096字符），防止配置异常导致切片失效。最后一块自动退出防止死循环。
- **EmbeddingService**：封装了两个外部 API——Ollama embedding 向量化 + ChromaDB 存储/检索。通过 DCL（双重检查锁）缓存 ChromaDB collection 的 UUID（ChromaDB 0.6+ 要求用 UUID 而非名称访问）。


- 将同场景所有其他 ACTIVE 版本降级为 DRAFT
- 将目标版本设为 ACTIVE

这样确保**一个场景只有一个活跃版本**，支持 A/B 测试场景下的版本切换。实体设计包含完整的 LLM 超参（temperature、topP、maxTokens）和 Few-Shot 示例字段。

### 涉及文件

| 层 | 文件路径 |
|----|---------|
| Controller | `controller/QAController.java` |
| Service | `service/QAService.java`（直接是类，无独立接口） |
| Mapper | `mapper/ChatLogMapper.java`、`mapper/PromptTemplateMapper.java` |
| Domain | `domain/ChatLog.java` |
| 辅助Service | `service/RetrievalService.java`、`service/PromptRenderService.java`、`service/EmbeddingService.java` |

### API 接口

| 方法     | 路径                            | 说明                    |
| ------ | ----------------------------- | --------------------- |
| POST   | `/api/qa/ask`                 | 同步RAG问答，返回完整回答        |
| POST   | `/api/qa/ask/stream`          | 流式RAG问答（SSE），逐token推送 |
| GET    | `/api/qa/chat-history`        | 查询会话对话历史              |
| DELETE | `/api/qa/delete-chat-history` | 删除会话对话历史              |

### 核心实现逻辑

**1. 同步问答 (QAService.ask) —— 完整6步流程：**

```
① 取提示词模板(PromptTemplateMapper.selectById)
   └── 动态选择topK: maxTokens<2048 就用3，否则用5

② 向量检索(RetrievalService.retrieve)
   ├── EmbeddingService.query(): 问题向量化 → ChromaDB语义检索 → 获取embedding ID列表
   └── 回MySQL查出完整文本块 → 按ChromaDB相关度顺序排列

③ Prompt渲染(PromptRenderService.render)
   └── 正则 {{variable}} 替换：{{question}} → 用户问题, {{context}} → 检索上下文

④ LLM调用(Spring AI ChatClient)
   └── .prompt().system(systemPrompt).user(fullPrompt).call().chatResponse()

⑤ 对话日志持久化(ChatLogMapper.insert)
   └── 记录sessionId、问题、检索块、完整prompt、回答、延迟、token估算

⑥ 返回QAResult {answer, sessionId, sources, latencyMs}
```

**2. 流式问答 (QAService.askStream) —— SSE 实现：**

```
Servlet AsyncContext(超时10分钟)
→ Executor后台线程执行
  ├── 检索 + Prompt渲染（同步）
  ├── 推送 SSE "meta" 事件（含sessionId和来源片段）
  ├── chatClient.prompt().stream().chatResponse()
  │   .publishOn(boundedElastic)
  │   .doOnNext(token → 写SSE "token"事件)
  │   .doOnComplete(写"done"事件 + 持久化ChatLog + asyncContext.complete())
  │   .doOnError(写"error"事件 + asyncContext.complete())
  │   .blockLast()
  └── AtomicBoolean safeComplete 保证complete()只调一次
```

**3. 支撑服务**

- **PromptRenderService**：正则 `\{\{(\w+)\}\}` 替换占位符，用 `Matcher.quoteReplacement()` 防止替换值中的特殊字符破坏正则语义
- **RetrievalService**：桥接ChromaDB语义检索结果与MySQL完整文本，保持ChromaDB返回的相关度顺序

### 核心实现逻辑

**1. 单条评测流程 (EvaluationService.evaluateSingle)：**

```
创建评测记录(RUNNING) → 保存RAG配置快照
→ 加载提示词模板
→ 检索上下文(RetrievalService.retrieve)
→ Prompt渲染 + ChatClient调用LLM生成回答
→ 计算检索客观指标(Java)：
  ├── contextRecall = 命中关键词数 / 总期望关键词数
  └── retrievalPrecision = 含关键词块数 / 总检索块数
→ LLM-as-Judge主观评分(调用LLM打分)：
  ├── answerRelevance(答案相关性 1-5)
  ├── contextFaithfulness(上下文忠实度 1-5)
  └── hallucinationScore(幻觉风险 1-5，越低越好)
→ 填充记录(含judgeRawResponse原始JSON) → 状态=COMPLETED
```

**2. LLM-as-Judge 工程设计要点：**

- 内嵌详细评分量规（Rubric）在系统提示词中，每个维度1-5分标准明确定义
- 使用低温 `temperature=0.1` 降低评分随机性
- **防御性JSON解析**：LLM可能返回 ` ```json...``` ` 代码块包裹的JSON，先正则清理再解析
- 存储 `judgeRawResponse` 用于事后偏差分析和评分校准
- 解析失败时所有分数返回0.0（而非抛异常）

**3. 批量评测（EvaluationService.runBatchEvaluation）：**

`@Async` 异步执行，遍历评测集中所有问题串行执行（避免LLM API限流）。每条包含2次LLM调用（生成+评分），完成后统计完成/失败数量。

**4. 两类评价指标的定位：**

| 类型 | 指标 | 计算方 | 用途 |
|------|------|--------|------|
| 客观 | 上下文召回率 | Java（关键词匹配） | 衡量检索是否遗漏了关键信息 |
| 客观 | 检索精准度 | Java（关键词匹配） | 衡量检索是否带回大量无关噪声 |
| 主观 | 答案相关性 | LLM打分 | 衡量生成回答是否切题 |
| 主观 | 上下文忠实度 | LLM打分 | 衡量回答是否基于知识库而非编造 |
| 主观 | 幻觉风险 | LLM打分 | 衡量编造事实的程度（越低越好）|

### 评测集管理（TestSetServiceImpl）

- 标准问答对：问题、参考答案、期望关键词（JSON数组）、难度(EASY/MEDIUM/HARD)、分类
- `listSetNames()` 用 MyBatis-Plus 的 select+groupBy 实现去重查询
- 按 setName 分组即视为同一批次

## 模块五：消融实验

### 涉及文件

| 层 | 文件路径 |
|----|---------|
| Controller | `controller/AblationController.java` |
| Service | `service/AblationService.java`（直接类） |
| Mapper | `mapper/AblationExperimentMapper.java`、`mapper/QaEvaluationRecordMapper.java`、`mapper/QaTestSetMapper.java` |
| Domain | `domain/AblationExperiment.java` |
| 依赖Service | `service/EvaluationService.java`（复用单条评测能力）|

### API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/ablation/experiments` | 创建消融实验（定义基线+变量） |
| GET | `/api/ablation/experiments` | 获取所有实验列表 |
| GET | `/api/ablation/experiments/{id}` | 获取实验详情与对比报告 |
| POST | `/api/ablation/experiments/{id}/run` | 启动实验（异步执行） |
| DELETE | `/api/ablation/experiments/{id}` | 删除实验 |

### 核心实现逻辑

**1. 组合生成 —— 递归笛卡尔积：**

```
输入示例：
  baseConfig = {promptTemplateId: 1, chunkSize: 512}
  variableConfigs = [{variable: "topK", values: [3, 5]}, {variable: "chunkSize", values: [256, 512]}]

生成过程（递归）：
  第0层: {} → 第1层追加 topK=3 → 第2层追加 chunkSize=256 → 输出组合1
                      → 第2层追加 chunkSize=512 → 输出组合2（remove）
         → 第1层追加 topK=5 → 第2层追加 chunkSize=256 → 输出组合3
                      → 第2层追加 chunkSize=512 → 输出组合4 （remove）

输出4个组合，每个带可读标签如 "topK=3, chunkSize=256"
```

**防组合爆炸**：默认上限50种组合，超出直接抛异常。`generateCombinations()` 预计算 `totalCombos = ∏(各变量取值数)` 先行校验。

**2. 异步执行 (AblationService.runExperiment)：**

```
@Async 执行:
更新实验状态=RUNNING
→ 解析配置 → 生成所有参数组合
→ 获取评测集活跃问题
→ 逐组合执行：
  ├── 生成groupTaskId（格式: abl-{实验ID}-{6位UUID}）
  ├── 对每个问题调 evaluationService.evaluateSingle(groupTaskId)
  └── 更新completedTasks进度 → updateById(exp)
→ 生成汇总报告 → 状态=COMPLETED
```

**3. 汇总报告 (AblationService.generateSummaryReport)：**

- 按 taskId 前缀匹配 (`likeRight "abl-{实验ID}-"`) 查询所有评测记录
- 分组计算各实验组的5项指标平均分：
  - avgAnswerRelevance、avgContextFaithfulness、avgHallucinationScore、avgContextRecall、avgRetrievalPrecision
- 输出各指标最优组：bestRelevance / bestFaithfulness / bestRecall / bestPrecision/（越高越好），bestHallucination（越低越好）
- 先从 `ragConfigSnapshot` JSON 中提取变量参数生成可读标签（过滤掉 `promptTemplateId`）

### 模块间依赖

消融实验本质上是**自动化评测的上层编排**——它不直接执行评测逻辑，而是通过 `EvaluationService.evaluateSingle()` 为每个参数组合生成唯一的 taskId，完成评测后再按 taskId 聚合对比。

### 涉及文件

| 层           | 文件路径                                     |
| ----------- | ---------------------------------------- |
| Controller  | `controller/TrainController.java`        |
| Service接口   | `service/TrainTaskService.java`          |
| ServiceImpl | `service/impl/TrainTaskServiceImpl.java` |

| Domain      | `domain/TrainTask.java`                  |
| Python端     | `python-train-side/`（FastAPI服务）          |

### API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/train/tasks` | 创建并启动微调任务 |
| GET | `/api/train/tasks` | 分页查询训练任务列表 |
| GET | `/api/train/tasks/{id}` | 获取任务详情（自动轮询最新状态） |
| POST | `/api/train/tasks/{id}/poll` | 手动轮询训练状态 |
| DELETE | `/api/train/tasks/{id}` | 删除训练任务 |

### 核心实现逻辑

**1. Java + Python 协同架构：**

```
┌─────────────┐     WebClient HTTP      ┌──────────────┐
│   Java端    │ ◄──────────────────────► │  Python端    │
│ (调度+持久化) │  POST /train            │ (QLoRA微调)  │
│             │  GET /train/{id}/status  │  FastAPI服务 │
│   MySQL     │                         │  GPU训练     │
└─────────────┘                         └──────────────┘
```

Java 端不写任何训练代码，只做两件事：**HTTP调度**（通过 WebClient 调用 Python 服务）和**状态持久化**（MySQL 记录训练进度和指标）。

**2. 创建并启动 (TrainTaskServiceImpl.createAndStart)：**

```
设置默认超参(loraRank=64, loraAlpha=16, lr=2e-4, epochs=3, batchSize=4)
→ 保存到MySQL(状态=PENDING, progress=0)
→ WebClient POST /train 调用Python服务
  ├── 成功: 记录pythonTaskId, 状态→TRAINING
  └── 失败: 保持PENDING, 记录errorMsg（任务不丢失，可后续重试）
→ updateById(task)
```

**3. 状态轮询 (TrainTaskServiceImpl.pollStatus)：**

```
调Python GET /train/{pythonTaskId}/status
→ 获取status字段 → mapStatus()映射状态：
  training/running → TRAINING
  completed/done/finished → COMPLETED
  failed/error → FAILED
→ 更新progress、metrics(训练指标JSON)、loraWeightPath
```

**4. 智能详情查询 (TrainTaskServiceImpl.getTaskDetail)：**

```
if (任务状态 == TRAINING) {
    先调 pollStatus() 获取最新状态  // 自动实时更新
}
return task
```

这使得 Controller 的 GET 接口天然具备获取实时训练进度和指标的能力。
