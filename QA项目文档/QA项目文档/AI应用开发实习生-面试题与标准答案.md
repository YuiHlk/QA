## Q1. 一句话说清楚你这个项目解决了什么问题？如果对面坐的是非技术背景的业务方，你怎么介绍？

**标准回答：**

对技术面试官：本项目解决了"如何让大模型基于企业内部私有知识文档进行精准、可量化评估的问答"问题，实现了一条从提示词管理 → 文档向量化 → 语义检索 → 增强生成 → 自动评测 → 消融实验优化的完整 AI 工程链路。

对业务方：公司内部有很多文档（产品手册、规章制度、技术方案等），员工遇到问题需要反复翻文档，效率很低。我们做了一个系统：员工用自然语言提问，AI 自动从公司文档中找到相关内容，生成准确回答。而且这个系统自己还能"给自己打分"，持续优化回答质量。

**回答要点：**
- 先讲痛点（私有知识难以被通用大模型利用）
- 再讲方案（RAG = 检索 + 生成）
- 最后讲价值（效率提升 + 可量化优化）

## Q2. 你项目的分层架构是怎么设计的（从 Controller 到 Mapper），每一层的职责边界在哪？为什么不能跨层调用？

**标准回答：**

```
Controller（控制器层）
  ↓ 调用
Service 接口（业务接口层）
  ↓ 调用
ServiceImpl（业务实现层）
  ↓ 调用
Mapper（数据访问层）
  ↓ 操作
Database（数据库）
```

各层职责：

| 层 | 职责 | 不该做的事 |
|----|------|-----------|
| Controller | 接收 HTTP 请求，参数校验，调用 Service，返回响应 | 不应该写业务逻辑、不应该直接操作数据库 |
| Service 接口 | 定义业务能力契约 | 不应该有实现代码 |
| ServiceImpl | 核心业务逻辑、事务管理、跨模块编排 | 不应该处理 HTTP 请求/响应对象 |
| Mapper | 数据库 CRUD 操作 | 不应该包含业务逻辑 |

**为什么不能跨层调用**（比如 Controller 直接调 Mapper）：

1. **职责混乱**：如果 Controller 直接调 Mapper，业务逻辑散落在各处，后续维护时不知道某个校验规则写在哪里
2. **难以复用**：Service 封装了可复用的业务逻辑，跨层调用会导致代码重复
3. **事务管理失效**：Spring 的事务是通过 Service 层的 AOP 代理实现的，跨过 Service 层事务注解不生效
4. **单元测试困难**：无法对业务逻辑进行独立测试，必须启动整个 Web 容器

**面试加分点**：可以提到你项目严格遵循了这个分层，代码目录结构（config/controller/service/domain/mapper）就是分层架构的物理体现。

## Q3. 项目中哪个模块是你觉得技术复杂度最高的？为什么？

**标准回答：**

RAG 检索增强生成模块技术复杂度最高，原因有三：

1. **链路最长**：涉及文档解析（PDFBox）→ 文本分块（滑动窗口）→ Embedding 向量化（调用 AI API）→ 向量存储（ChromaDB）→ 语义检索 → Prompt 渲染 → LLM 生成 → 对话记录持久化，任何一个环节出错都会导致最终回答质量下降

2. **AI 特有的不确定性**：传统软件开发输入输出是确定的（1+1 一定等于 2），但 RAG 系统中——
   - 分块策略影响检索质量（切错位置可能丢失关键信息）
   - Embedding 模型选择影响语义匹配准确度
   - LLM 生成可能产生幻觉（回答看起来很合理，实际跟文档内容无关）
   - 这种不确定性要求工程设计上做大量兜底和验证

3. **性能与体验的平衡**：大文档处理耗时长（解析 + 分块 + 批量向量化），必须用异步处理保证用户体验（上传即返回），同时又要让用户知道处理进度，涉及线程池配置、状态机设计、异常重试等工程考量

**面试提示**：回答这个问题时不要泛泛而谈，要结合你的实际代码讲——比如你的 `@Async` 异步处理管道具体怎么设计的。

# 二、RAG 核心链路

---

## Q4. 从用户上传一个 PDF 文档，到用户基于这个文档提问得到回答，整个数据流是怎样的？每一步发生了什么？

**标准回答：**

这是两个阶段，我先分开讲，再串起来。

### 阶段一：文档入库（离线处理，异步执行）

```
上传 PDF → 保存元信息(status=PROCESSING) → 立即返回成功
                                          ↓
                                    @Async 异步线程：
                                    PDFBox 提取文本
                                          ↓
                                    滑动窗口分块
                                    (chunkSize=512, overlap=50)
                                          ↓
                                    逐块调用 Embedding API
                                    文本 → 1536维向量
                                          ↓
                                    向量+元数据批量写入 ChromaDB
                                          ↓
                                    更新文档状态 status=READY
```

### 阶段二：用户问答（在线处理，同步执行）

```
用户提问
  → 问题文本调用 Embedding API → 问题向量
  → ChromaDB.topK=5 语义检索 → 5个最相关文档片段
  → 拼接检索上下文 + 系统提示词 + 用户问题
  → 调用 LLM（如 GPT-4o）生成回答
  → 保存对话记录(USER消息 + ASSISTANT消息)
  → 返回回答给前端
```

**关键设计决策**：阶段一异步、阶段二同步。因为上传文档用户不需要立即看到结果，可以"提交后不管"；但提问时必须同步等待回答，这是交互式体验。

**回答要点：**
- 画图比说文字更清晰，面试时可以在白板上画数据流
- 强调异步设计的原因
- 提到每一步用到的具体技术（PDFBox、Embedding API、ChromaDB、LLM）

## Q5. 文档分块时你用了滑动窗口，chunkSize 和 overlap 怎么定的？如果 chunkSize 设为 200 或 2000，分别会有什么问题？

**标准回答：**

我项目中的默认配置是 chunkSize=512 字符，overlap=50 字符（约 10% 的重叠率）。

**chunkSize 太小（如 200）的问题：**
- 每个块包含的语义信息不完整，一段完整论述可能被切到 3-4 个块里
- 检索时匹配到的块缺乏完整上下文，LLM 拿到的信息是碎片化的，生成质量下降
- 分块数量暴增，向量存储和检索成本增加

**chunkSize 太大（如 2000）的问题：**
- 每个块包含的信息太杂，一个块可能混合了多个主题
- 检索精度下降——用户问题匹配到某个块后，这个块里 80% 的内容可能跟问题无关（噪声）
- LLM 的 context window 被无效信息占据，真正有用的部分被稀释

**overlap 的作用：**
- 防止关键信息被切在块边界。比如："产品的核心优势在于"这句话在块的结尾，而"高并发低延迟"这句话在下一个块的开头
- 如果没有 overlap，检索"产品优势"时两个块各匹配一半，分数都不高，可能都排不进 topK

**选值方法论**：chunkSize 的选择跟文档类型相关——技术文档偏短（256-512），长文报告偏长（1024-2048）。我项目中通过消融实验模块系统地对比了不同 chunkSize 对最终评测指标的影响，找到了最适合知识库文档的值。

**面试加分点**：提到"消融实验验证"说明你不是拍脑袋定的参数，是有数据支撑的。

## Q6. Embedding 向量化是什么意思？"今天天气真好"和"今天天气不错"的向量距离，跟"今天天气真好"和"资产负债表分析"的向量距离，哪个更近？为什么？

**标准回答：**

**Embedding 是什么：**
Embedding 是将文本映射到高维向量空间（比如 1536 维）的数学表示。本质上就是把一段文本的语义信息压缩为一组浮点数 `[0.23, -0.45, 0.78, ...]`，使得语义相近的文本在这个向量空间中位置靠近。

可以类比理解：把每个文本想象成地图上的一个点，地图的经纬度坐标就是它的向量。语义相近的文本，就像地理位置靠近的两个城市。

**距离比较：**
"今天天气真好"和"今天天气不错"的向量距离更近。两者的核心语义都是"天气好"，只是表达方式略有不同。

"资产负债表分析"讨论的是完全不同的领域（财务），语义上离"天气"很远，所以向量距离很大。

**为什么 Embedding 能做到这一点：**
Embedding 模型（如 OpenAI 的 text-embedding-3-large）是在海量文本上训练出来的，训练过程中模型学会了"哪些词语和上下文经常一起出现"。"天气好"和"天气不错"在训练语料中经常是近义的上下位词或同义表达，它们的向量被训练得很接近。

**在 RAG 中的应用：**
用户的问题是"如何查看月度销售报表"，Embedding 模型会把这个问题向量化，然后在向量空间中找到最接近的文档片段——哪怕文档里写的是"销售数据月度查看方法"，关键词不完全一样，但语义相近，依然能匹配上。这就是向量检索优于传统关键词搜索的原因。

## Q7. 用户问题怎么找到相关的文档片段？向量数据库的检索和传统 MySQL 的 LIKE 模糊搜索有什么本质区别？

**标准回答：**

**向量检索流程：**
```
用户问题 → Embedding API → 问题向量
问题向量 → ChromaDB → 余弦相似度计算 → topK 个最近邻向量 → 对应文档片段
```

ChromaDB 内部使用近似最近邻搜索算法（ANN），不需要把问题向量和所有文档向量逐一比较，而是通过索引结构（如 HNSW 图索引）快速定位最近的几个向量。

**本质区别：**

| 维度 | MySQL LIKE 模糊搜索 | 向量语义检索 |
|------|---------------------|-------------|
| 匹配方式 | 字符串子串匹配 | 语义相似度计算 |
| 例子 | `WHERE content LIKE '%销售报表%'` | 问题向量与文档向量的余弦相似度 |
| 同义词 | "购买"搜不到"采购" | "购买"能找到"采购"（语义相近） |
| 跨语言 | 中文搜不到英文 | 多语言 Embedding 模型可跨语言 |
| 排序 | 按时间/字段排序 | 按语义相关性排序 |
| 理解能力 | 零（纯字符匹配） | 强（理解意图和语义） |

**实际例子：**
假设用户问"怎么查上个月的销售额？"

- MySQL LIKE：搜不到，因为文档里写的可能是"月度营收数据查询方法"，没出现过"销售额"这三个字
- 向量检索：能命中，因为"上个月+销售额+查"和"月度+营收+查询"语义高度相近，向量距离很近

**什么时候用哪个：**
- 精确匹配场景（搜索订单号、人名）→ LIKE 或精确索引
- 语义理解场景（知识问答、文档搜索）→ 向量检索
- 实际生产环境常用混合检索（Hybrid Search）：向量检索 + 关键词检索，加权融合排序

## Q8. 如果你的 RAG 系统产出了一个错误回答，你从哪几个环节排查问题？

**标准回答：**

我的排查思路是从链路末端往前逐环节排查：

**第 1 步：检查 LLM 生成环节**
- 看 LLM 拿到的完整 Prompt 是否正确（系统提示词 + 检索上下文 + 用户问题）
- 如果 Prompt 本身就有问题，那 LLM 生成错误是必然的
- 调整 Temperature 参数（太高会导致模型"发散"，产生随机内容）

**第 2 步：检查检索环节**
- 看检索到的 topK 个文档片段是否跟用户问题真正相关
- 可能的问题：检索到了不相关的片段（语义漂移），或者相关片段根本没被检索到
- 对策：调整 Embedding 模型、增大 topK、优化分块策略

**第 3 步：检查 Embedding 向量化环节**
- Embedding 模型的选择直接影响语义匹配质量
- 中文文档用了英文优化的 Embedding 模型 → 效果差
- 解决：换用专门支持中文的 Embedding 模型（如 text-embedding-3-large, bge-large-zh 等）

**第 4 步：检查文档分块环节**
- 分块策略不合理，导致检索到的片段缺乏完整语义
- chunkSize 太大→噪声多，太小→信息碎片化
- 解决：通过消融实验找到最优 chunkSize

**第 5 步：检查原始文档质量**
- 原始文档本身就有错误（过时信息、格式混乱、扫描件 OCR 识别错误）
- 问题不在技术上，需要从文档管理流程上解决

**排查方法论：**
发现问题后不要凭直觉改参数，而是用 LLM-as-Judge 评测模块系统性地衡量优化效果——改了分块策略后，评测分数是涨了还是跌了，用数据说话。

**面试加分点**：这个回答体现了你的"排查能力"——不是盲目 debug，而是有方法论、有先后顺序、用数据验证。

# 三、LLM-as-Judge 评测体系

---

## Q9. 你项目的评测体系分了"客观指标"和"主观指标"两类，为什么要这样区分？每一类各解决了什么问题？

**标准回答：**

**客观指标**（Java 代码计算）：
- 检索精准度（Precision）：检索到的片段中，有多少是真正相关的
- 上下文召回率（Recall）：真正相关的片段中，有多少被检索到了

这两个指标完全由数据决定，不依赖 AI 判断，结果是确定且可复现的。

**主观指标**（LLM 打分）：
- 答案相关性：回答是否切题
- 上下文忠实度：回答是否基于给定文档（不是编造的）
- 幻觉风险：回答中是否存在无依据的信息

这三个指标没有标准答案，依赖 AI 的"理解"和"判断"。

**为什么要分两类：**

1. **客观指标验证检索质量，主观指标验证生成质量**。RAG 系统 = 检索（Retrieve）+ 生成（Generate），两部分都需要独立评估。检索再好，生成环节胡编乱造也没用；生成能力再强，检索不到相关内容也是"巧妇难为无米之炊"。

2. **客观指标可复现、可信度高**：Precision 和 Recall 是基于标注数据计算的确定值，明天再算还是同一个数。这是评测体系的"锚点"。

3. **主观指标覆盖客观指标无法衡量的问题**：比如"回答是否流畅自然"、"是否存在微妙的事实扭曲"，这些目前只能靠 AI 或人工来评判。

4. **交叉验证**：如果一个实验组合的客观指标很好（检索到了正确文档）但主观指标很差（LLM 没有正确利用检索到的信息），说明问题出在 Prompt 设计或 LLM 选择上，而不是检索环节。

## Q10. LLM-as-Judge 让 AI 评价 AI，你如何保证评判结果是可信的？如果 LLM 给出的评分不稳定（同一道题打分忽高忽低），你怎么处理？

**标准回答：**

**可信度保障措施：**

1. **结构化输出约束**：Judge Prompt 要求 LLM 严格输出 JSON 格式 `{"relevance": X, "faithfulness": X, "hallucination": X}`，便于程序化解析，避免自由文本分析中的歧义

2. **精细化评分标准**：Prompt 中明确定义了 1-5 分的含义——不是"你觉得几分就几分"，而是 1 分="完全无关"，5 分="高度相关且精准"，有效降低自由度

3. **客观指标交叉验证**：如果 LLM 打分显示"上下文忠实度"很高（4-5 分），但客观 Precision 很低（检索到的文档基本不相关），这说明 LLM 的打分有问题——检索都不对，怎么可能忠实于正确的文档？

4. **Temperature=0**：打分时把 Temperature 设为 0（或很低如 0.1），大幅降低随机性

5. **多次打分取平均**：必要时对同一条记录进行 3-5 次独立打分，取平均值，降低单次波动影响

**评分不稳定的处理：**

1. 检查 Temperature 设置，确定打分时设为了接近 0
2. 优化 Judge Prompt——评分标准是否足够具体、歧义是否太大
3. 换用更强的模型做 Judge（如用 GPT-4 给 GPT-3.5 的输出打分，因为更强的模型判断力更好）
4. 如果某类问题评分始终不稳定，说明这个维度的定义需要细化——比如把"相关性"拆成"主题相关"和"细节准确"两个子维度

**核心认知**：LLM-as-Judge 不是完美的，它是"足够好的自动化近似"，用于快速迭代和横向对比，最终的关键决策仍需人工抽检。

## Q11. 你的评测提示词（Judge Prompt）是怎么设计的？为什么选了答案相关性、上下文忠实度、幻觉风险这三个维度而不是其他维度？

**标准回答：**

**Judge Prompt 设计结构：**

```
1. 角色设定    → "你是一个专业的AI回答质量评审专家"
2. 输入信息    → 用户问题 + 参考文档 + AI回答
3. 评分维度    → 三个维度，每个维度1-5分，有明确标准
4. 输出格式    → 严格 JSON {"relevance": X, "faithfulness": X, "hallucination": X}
```

**为什么选这三个维度：**

这是从 RAG 系统最常出现的三类问题倒推设计的：

| 常见问题 | 对应评测维度 | 典型表现 |
|----------|-------------|----------|
| 答非所问 | 答案相关性 | 用户问"怎么退货"，AI 回答"如何下单" |
| 胡编乱造 | 上下文忠实度 | 文档没写"7天无理由"，AI 自己加上去了 |
| 看似有理实则虚构 | 幻觉风险 | 编造不存在的产品功能、虚构数据 |

这三个维度是 RAG 评测领域最受认可的基础维度集，来自于 RAGAS（RAG Assessment）论文的评估框架。它们共同覆盖了 RAG 回答质量的三个侧面：
- **相关性**：回答对不对路
- **忠实度**：回答有没有依据
- **幻觉**：回答有没有瞎编

**不选其他维度的原因：**
- "回答流畅度"：现代 LLM 生成的文本语法基本都不会出错，这个维度区分度低
- "回答完整性"：与"相关性"有重叠，且难以客观定义"完整"的标准
- "有毒有害检测"：与知识问答场景相关性不强，更适用于对话机器人

# 四、消融实验

## Q12. 消融实验的"控制变量法"思想在你这项目中具体是怎么落地的？你设计了哪几个变量、每个变量有哪些候选值？

**标准回答：**

**控制变量法的落地方式——笛卡尔积实验设计：**

不同于科研中的"一个一个改"（改动成本高），利用软件工程的自动化能力同时测试所有组合：

```java
// 伪代码展示设计思路
chunkSize:  [256, 512, 1024]        // 3个候选
topK:       [3, 5, 7]              // 3个候选
promptId:   [模板v1, 模板v2]        // 2个候选
modelName:  [gpt-4o, deepseek-v3]  // 2个候选

// 笛卡尔积生成 3×3×2×2 = 36 个实验组合
// 每个组合在同一个测试集上执行，记录评测指标
// 最终产出对比报告
```

**为什么选这四个变量：**

| 变量 | 影响环节 | 为什么需要实验 |
|------|---------|---------------|
| chunkSize | 文档分块→检索质量 | 影响检索到的信息粒度 |
| topK | 检索→上下文拼接 | 影响给 LLM 的上下文量和噪声量 |
| promptId | Prompt 渲染→LLM 理解 | 不同提示词模板效果差异很大 |
| modelName | LLM 生成 | 不同模型的能力和价格差异显著 |

**实验执行流程：**
1. 用户选择变量维度和候选值
2. 系统生成笛卡尔积实验列表
3. 逐条自动执行（同一测试集上跑）
4. 每条记录自动评测（客观 + LLM 打分）
5. 生成对比报告：按变量分组聚合，自动标注每项指标最优值

**实际价值**：比如经过实验发现 chunkSize=512 比 256 在检索精准度上高 15%，Prompt v2 在幻觉风险上显著优于 v1——这些就是数据驱动的优化决策依据。

## Q13. 多个变量的笛卡尔积会导致实验数量爆炸，你的安全限制策略是什么？如果公司要求 6 个变量、每个 5 个候选值，你怎么办？有没有比全量笛卡尔积更高效的设计？

**标准回答：**

**当前安全限制策略：**
设置了最大实验组合数上限（如 100），超过后拒绝执行并提示"实验组合过多，请减少变量维度"。

**6 变量 × 5 候选 = 15,625 个组合怎么处理：**

盲目笛卡尔积显然不可行（时间和 API 费用都扛不住），改用以下策略：

**方案一：分阶段实验（推荐）**

不要一次性测所有变量，分两到三轮：

```
第一轮：固定其他变量为默认值，只测 chunkSize（5 个候选，共 5 次实验）
  → 确定最优 chunkSize = 512

第二轮：固定 chunkSize=512，只测 topK（5 个候选，共 5 次实验）
  → 确定最优 topK = 5

... 以此类推
```

6 个变量各 5 个候选，分 6 轮，每轮 5 次实验 = 共 30 次。从 15,625 降到 30。

**方案二：正交实验设计（L_n 正交表）**

科学实验中常用的统计方法，不是测所有组合，而是选取"最具代表性"的部分组合，用统计学方法推断各变量的影响。比如 L25(5^6) 正交表可以用 25 次实验覆盖 6 因子 5 水平的设计。

**方案三：贝叶斯优化（Bayesian Optimization）**

适合连续参数（如 Temperature 0.1-1.0），用概率模型预测哪些参数组合最值得尝试，每轮选最有"探索价值"的点测试，收敛到最优解的速度远快于网格搜索。

**方案四：优先测关键变量**

如果对系统有一定先验知识，可以优先测影响最大的变量。比如已知 Prompt 版本质量差异远大于 chunkSize 的差异，那就先把 Prompt 的最优版本确定下来，再调其他参数。

**核心思想**：消融实验的目标不是"把所有可能都测一遍"，而是"花最少的实验次数找到最佳配置"。这是一个"实验设计（DoE）"的问题，而不是暴力穷举的问题。

## Q14. 你项目的架构里 Java 和 Python 各自承担什么角色？为什么不让 Java 直接做模型微调？

**标准回答：**

**角色分工：**

| | Java (Spring Boot) | Python (FastAPI) |
|------|------|------|
| 定位 | 主服务，业务编排 | 专用 AI 子服务 |
| 职责 | HTTP API、数据库 CRUD、RAG 链路、任务调度 | 模型微调、深度学习训练 |
| 端口 | 8080 | 8002 |
| 通信 | 作为调用方（WebClient） | 作为被调用方（REST API） |

**为什么不让 Java 做模型微调：**

1. **生态问题**：Python 是 AI/ML 领域的"母语"。PyTorch、transformers、peft、bitsandbytes 这些库都是 Python 生态的。Java 没有等价的生产级替代品。

2. **人才问题**：做模型微调的算法工程师精通 Python，如果他们写的训练代码需要嵌入 Java 项目，协作成本极高。Python 子服务独立部署后，算法工程师改训练逻辑不需要动 Java 代码。

3. **部署隔离**：模型微调非常耗资源（GPU 显存、计算时间），如果跟主服务部署在同一进程，训练时会把整个 Java 服务拖垮。独立成子服务，资源隔离，互不影响。

4. **解耦与可替换性**：微调服务有标准的 REST API（`POST /train`, `GET /train/{id}/status`），以后换训练框架、换模型，只要 API 不变，Java 侧不需要改任何代码。

**本质**：这是微服务架构中"异构服务协作"的经典实践——每个语言做自己最擅长的事。

## Q15. Java 侧用 WebClient 调用 Python 服务，WebClient 和 RestTemplate 有什么区别？为什么选 WebClient？

**标准回答：**

**RestTemplate：**
- 基于 Servlet API，每个请求占用一个线程
- 同步阻塞模式：发请求 → 等响应 → 处理结果，等的时候线程干等着什么事都不做
- Spring 5 起被标记为"维护模式"，不再添加新功能

**WebClient：**
- 基于 Reactive 编程模型，支持异步非阻塞 I/O
- 发请求后不阻塞线程，可以继续处理其他任务，响应回来时通过回调处理
- Spring 官方推荐的现代 HTTP 客户端

**场景对比：**

同步阻塞（RestTemplate）：
```
[线程1] POST 请求到 Python → 等待 10 分钟训练... → 收到响应
       ↑ 线程一直占着，什么都不做
```

异步非阻塞（WebClient）：
```
[线程A] POST 请求到 Python → 线程释放，去处理其他请求
                                 ↓ 10 分钟后
[线程B] 收到响应，回调处理训练结果
```

**我项目中的实际选择原因：**
1. 微调任务耗时很长（几十分钟到几小时），如果同步阻塞会一直占着 Tomcat 线程，高并发下线程池很快耗尽
2. WebClient 天然适合"发出请求 + 异步回调更新任务状态"的流程
3. 跟 Spring 的 `@Async` + `CompletableFuture` 配合使用，代码更简洁
4. Spring 生态长期方向——未来 Spring Boot 版本中可能移除 RestTemplate

**代码示例：**
```java
// WebClient 异步调用，配合 @Async 实现非阻塞等待
webClient.post()
    .uri("http://localhost:8002/train")
    .bodyValue(params)
    .retrieve()
    .bodyToMono(TrainResponse.class)
    .block();  // 在@Async线程中阻塞等待，不占用主线程
```

## Q16. QLoRA 微调中的"4bit 量化"和"LoRA 低秩适配"分别解决什么问题？为什么微调 7B 级别的模型能在消费级显卡上跑？

**标准回答：**

**资源瓶颈分析：**

一个 7B 参数的模型（如 Qwen2-7B-Instruct）：
- 全精度（FP32）：每个参数 4 字节 → 7B × 4B = 28GB 显存
- 训练时还需存储优化器状态（Adam 需要额外 8 字节/参数）、梯度（4 字节/参数）
- 全量微调实际需要：28GB（模型）+ 56GB（优化器）+ 28GB（梯度）= 112GB 显存
- 这需要 A100 80GB × 2 = 远超消费级硬件

**4bit 量化解决模型大小问题：**

```
FP32: 每个参数 32bit（4字节），7B模型 = 28GB
4bit: 每个参数 4bit（0.5字节），7B模型 ≈ 3.5GB
```

用 BitsAndBytes 库的 NF4（NormalFloat4）量化格式，把模型权重从 32bit 压缩到 4bit，显存占用降到原来的 1/8。同时采用 4bit 存储 + 16bit 计算的混合精度——存储用 4bit 省显存，计算时反量化为 16bit 保证精度。

**LoRA 低秩适配解决训练参数量问题：**

LoRA（Low-Rank Adaptation）的核心思想：不修改原模型参数，在 Transformer 的注意力层旁路插入小的可训练矩阵。

```
原模型权重矩阵 W (d×d, 被冻结)
LoRA: A (d×r) × B (r×d)，其中 r << d（如 r=8, d=4096）

W' = W + A × B

训练参数 = d×r + r×d = 2dr = 2 × 4096 × 8 = 65,536 个参数  ← 只更新这部分
原参数 = d×d = 4096 × 4096 = 16,777,216 个参数              ← 冻结不变

可训练参数占比: 65,536 / 16,777,216 ≈ 0.4%
```

**最终效果：一块 RTX 4060（16GB 显存）就能微调 7B 模型：**

```
4bit 量化模型:  ~3.5GB
LoRA 适配器:    ~0.1GB（可训练参数极少）
优化器状态:     ~0.4GB
梯度存储:       ~0.1GB
总计:           ~4-5GB ✓ 完全在 16GB 显存内
```

**面试时要讲清楚的关键点：**
- 4bit 量化解决的是"模型放不放得下"的问题（存储成本）
- LoRA 解决的是"训练训不训得动"的问题（计算成本）
- 两者结合，才能让 7B 模型的微调从"需要 2 张 A100"变成"一张消费级显卡搞定"

## Q17. 你项目中自定义了异步线程池，核心参数怎么配置的？如果上传一个 50MB 的大文档，系统怎么保证不阻塞其他用户的请求？

**标准回答：**

**线程池参数配置：**

```java
@Bean("taskExecutor")
public ThreadPoolTaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(5);      // 核心线程数：常驻 5 个线程
    executor.setMaxPoolSize(10);       // 最大线程数：高峰期最多 10 个
    executor.setQueueCapacity(100);    // 任务队列：最多排队 100 个任务
    executor.setThreadNamePrefix("async-doc-");  // 线程命名，便于排查
    executor.setRejectedExecutionHandler(
        new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：调用者执行
    );
    executor.initialize();
    return executor;
}
```

**线程池工作流程：**

```
新任务提交
  ↓
核心线程有空闲？ → 是 → 核心线程执行
  ↓ 否
队列有空位？ → 是 → 放入队列等待
  ↓ 否
线程数 < maxPoolSize？ → 是 → 创建新线程执行
  ↓ 否
执行拒绝策略（CallerRunsPolicy → 由提交任务的线程自己执行）
```

**50MB 大文档不阻塞其他用户的保障：**

1. **异步处理**：上传接口只做文件接收 + 元信息保存（耗时毫秒级），立即返回成功。文档解析、分块、向量化等耗时操作全部在 `@Async` 线程池中执行。Tomcat 的请求处理线程被快速释放去服务其他用户。

2. **线程隔离**：文档处理用的是自定义线程池 `taskExecutor`，跟 Tomcat 的请求处理线程池完全隔离。就算 `taskExecutor` 的线程全被大文档占满，Tomcat 依然能接收新用户的请求。

3. **状态反馈**：文档表有 `status` 字段（PROCESSING → READY / FAILED），用户可以通过定时轮询或手动刷新查看处理进度。不会让用户对着一个"转圈"的界面干等。

4. **可观测性**：线程名带前缀 `async-doc-`，通过 jstack 或日志可以快速定位是哪个文档在处理。

**为什么不设更大的线程数：**
线程多了竞争上下文切换的开销会超过并发带来的收益。文档处理是 CPU + IO 密集型（解析 + 网络调用 Embedding API），5-10 个线程是比较合理的值。具体可以通过压测数据调优。

## Q18. 你的 `application.yml` 中 API Key 是怎么管理的？如果在团队协作中，你不小心把 API Key 提交到 Git 了怎么办？

**标准回答：**

**安全管理策略：**

1. **环境变量注入**：yml 中只写占位符，运行时从环境变量读取
```yaml
ai:
  api-key: ${AI_API_KEY}    # 不写死真实值
```

2. **`.env.example` 模板文件**：提供配置模板（只有 key 名，没有真实值），团队成员基于模板创建自己的 `.env` 文件

3. **`.gitignore` 排除敏感文件**：
```
.env          # 包含真实 API Key，不进版本控制
*.log          # 日志可能包含敏感信息
data/          # 数据库文件
```

**API Key 泄露后的补救措施：**

1. **立即到 API 平台吊销旧 Key，生成新 Key**——这是必须做的！一旦 Key 进了 Git 历史，所有有仓库访问权限的人（包括后来加入的）都能从历史记录里找到它

2. **清理 Git 历史**：
```bash
# 使用 git filter-branch 或 BFG Repo-Cleaner 从全部历史中删除
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch .env" \
  --prune-empty --tag-name-filter cat -- --all
git push --force
```

注意：如果仓库已经被 fork 了，清理自己的仓库也没用，fork 出去的人仍然有那份历史。所以**最关键的永远是第一步（吊销旧 Key）**。

3. **团队通知**：告知所有成员旧 Key 已泄露，需更新本地的 `.env`

**预防措施**：
- 使用 `.gitignore` 全局模板
- 配置 Git pre-commit hook 自动扫描敏感信息（如 git-secrets）
- 代码审查时特别注意 `.env` 或 yml 文件是否包含硬编码的 Key

## Q19. Docker Compose 编排了 5 个服务，服务之间的启动顺序和网络通信是怎么设计的？`depends_on` 能保证 MySQL 完全就绪后再启动后端吗？

**标准回答：**

**启动顺序设计：**

```
1. MySQL（基础依赖）    —— 最先启动
2. ChromaDB（基础依赖）  —— 与 MySQL 并行启动
3. Backend（依赖 MySQL + ChromaDB）—— 基础服务就绪后启动
4. Frontend（依赖 Backend）—— 后端就绪后启动
5. Python Train（独立服务）—— 独立启动
```

**容器间网络通信：**

Docker Compose 自动创建了一个自定义 bridge 网络，所有服务在这个网络中可以直接用**服务名作为主机名**通信：

```yaml
# 容器内访问示例
jdbc:mysql://mysql:3306/qa_platform     # 不是 localhost，而是服务名 mysql
http://chromadb:8000                     # 用服务名 chromadb
```

**`depends_on` 的局限：**

```yaml
backend:
  depends_on:
    - mysql
```

`depends_on` **只能保证容器先启动，不能保证服务已就绪**。MySQL 容器启动 ≠ MySQL 能接受连接（可能需要初始化数据、恢复事务日志等），后端连不上会报错。

**正确做法——`healthcheck` + `condition`：**

```yaml
mysql:
  healthcheck:
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
    interval: 10s
    timeout: 5s
    retries: 5

backend:
  depends_on:
    mysql:
      condition: service_healthy    # 关键：等 MySQL 健康检查通过才启动
```

这样后端只会在 MySQL 真正能接受连接时才启动，避免了启动顺序导致的连接失败。

**后端的容错兜底**：即便做了健康检查，也应该在应用层加数据库连接重试机制（Spring Boot 默认就有），双重保障。

## Q20. 假设你的 RAG 系统上线后，用户反馈"问同一个问题，有时回答很好，有时完全不对"，你会从哪些方向排查？跟你的消融实验模块有什么关系？

**标准回答：**

**排查方向（从最可能到最不可能）：**

**1. LLM 的 Temperature 参数设置**
- Temperature 越高，相同 Prompt 的输出差异越大
- 排查：检查当前 Prompt 模板的 Temperature 配置，知识问答场景应该接近 0（如 0.1-0.3）
- 如果设置过高（如 0.8-1.0），同一个问题多次回答确实会差很多

**2. 检索结果不稳定**
- 同一个问题的 Embedding 向量应该是确定的，但不同的提问方式会导致不同的检索结果
- 排查：记录两次提问的具体措辞和各自的检索 topK 结果，对比是否一致
- 问题可能出在：Embedding 模型对某些近义表达不够鲁棒、或者某些文档片段覆盖的主题模糊容易被误匹配

**3. 多个文档包含矛盾信息**
- 比如文档 A 说"退货政策为 7 天"，文档 B 说"退货政策为 15 天"（两者针对不同品类）
- 两次提问检索到的 topK 片段不同（一个命中文档 A 多，一个命中文档 B 多）
- LLM 严格依据检索到的上下文生成 → 矛盾的结果

**4. LLM 本身的随机性**
- 即使 Temperature=0，部分模型的输出仍然会有微小的不确定
- 调用的 API 服务侧可能有负载均衡到不同的模型版本

**跟消融实验的关系：**

这正好体现了消融实验的价值——不是靠猜测来定位问题，而是：
1. 在消融实验中加入"重复测试"维度：同一个组合跑 5 次，看评测分数的方差
2. 如果方差大，说明这个配置组合的稳定性差，需要调整（降低 Temperature、优化 Prompt 约束）
3. 通过对比不同配置组合的稳定性和效果，找到既好又稳的参数组合
4. 这是"可观测的 AI 工程"思维——每个问题都应该能追溯到具体的指标变化

## Q21. 你项目的向量数据库用了 ChromaDB。如果公司的文档量从 1000 份增长到 100 万份，ChromaDB 还够用吗？你考虑过哪些替代方案？

**标准回答：**

**ChromaDB 的适用边界：**

ChromaDB 适合中小规模场景（10 万-50 万向量以内），它基于内存索引 + SQLite 持久化，优点是轻量、免费、部署简单。但当规模到百万级时，面临的问题：
- 内存索引膨胀，检索延迟上升
- 单节点无分布式能力
- 缺乏高级索引优化（如 IVF、PQ 等近似索引算法）

**百万级文档的替代方案：**

| 方案 | 适用规模 | 特点 | 成本 |
|------|---------|------|------|
| **Milvus** | 亿级向量 | 分布式架构、多种索引算法（IVF、HNSW、DiskANN）、支持混合检索 | 开源免费，运维成本高 |
| **Pinecone** | 无上限 | 全托管 SaaS、零运维、自动扩缩容 | 按量付费，大规模时成本较高 |
| **Weaviate** | 千万-亿级 | 开源、自带向量化模块、支持 Hybrid Search | 开源免费或云服务付费 |
| **Qdrant** | 千万-亿级 | Rust 编写、高性能、支持量化压缩 | 开源免费或云服务付费 |
| **Elasticsearch + 向量插件** | 亿级 | 已有 ES 基础设施的团队首选，向量+全文混合检索 | 开源免费 |

**升级路径（渐进式）：**

```
当前：ChromaDB（开发/中小规模）
  ↓ 数据量 > 50万
迁移：Milvus / Qdrant（自建分布式集群）
  ↓ 团队运维能力不足
迁移：Pinecone / Zilliz Cloud（全托管云服务）
```

**选型核心考量：**
1. 数据规模 = 文档量 × 每文档平均分块数 × 向量维度
2. QPS 要求（检索并发量）
3. 团队运维能力（自建开源 vs 全托管云服务）
4. 是否需要混合检索（向量 + 关键词混合排序）

**面试加分点**：如果你说出"在你的架构里，向量数据库的调用封装了接口（如 VectorStore 接口），底层换数据库不需要改业务代码"，这体现了良好的抽象设计能力。

## Q22. 回顾整个项目，如果再给你两个月时间，你最想在哪个模块上做深入优化？为什么？

**标准回答：**

我最想深入优化的是 **RAG 检索模块**，原因如下：

**当前方案的局限性：**

1. **单一粒度分块**：所有文档用同一套 chunkSize，但不同类型文档（短问答 vs. 长技术文档）的最优分块策略不同。应该支持自适应分块。

2. **粗粒度检索**：当前是一次性检索 topK 个片段后拼接，但更好的做法是"父文档检索"——先检索到相关小块，再把小块所在的完整段落（大块）一起给 LLM，保留完整的上下文。

3. **缺乏检索重排序**：检索到的 topK 个片段可能包含"看起来相关但其实不相关"的，应该加一个重排序（Rerank）步骤，用更强的模型对初始检索结果精排，只保留真正相关的。

4. **单轮检索**：用户问题直接检索，但如果问题本身比较模糊，可以先让 LLM 改写或拆解问题（Query Rewriting / Decomposition），再进行检索。

**优化方案（按优先级排序）：**

**Phase 1 — 加入 Rerank 重排序：**
```
初始检索(K=20) → Rerank 模型精排 → 取前 Top5 → 拼接上下文 → LLM 生成
```
用一个更强但更慢的模型对 20 个候选片段重排序，既保证了召回（初始检索用较大 K 值），又保证了精度（只给 LLM 最相关的 5 个）。

**Phase 2 — 混合检索（Hybrid Search）：**
```
向量语义检索结果 + 关键词 BM25 检索结果 → 加权融合排序
```
解决单纯向量检索在某些场景下"语义接近但其实不相关"的问题。比如有人问"苹果手机的价格"，向量检索可能把"苹果水果的价格"排得很高，加上 BM25 关键词匹配后可以纠正。

**Phase 3 — 多粒度索引（父文档检索）：**
```
小块检索（用于精确匹配）+ 大块上下文（用于 LLM 理解完整语义）
```
检索到的每个小块，同时返回它所在的完整段落。

**为什么这个方向最值得投入：**
- RAG 系统"天花板在检索"——检索质量直接决定了最终回答的上限，LLM 再强也无法基于错误信息生成正确回答
- 每一个优化都可以通过现有的 LLM-as-Judge 评测模块量化验证（形成了"优化→评测→再优化"的闭环）
- 这些优化是通用的，跟具体业务场景无关
